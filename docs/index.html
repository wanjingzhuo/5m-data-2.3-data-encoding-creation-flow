Data Encoding & Flow: A Beginner's Guide

This lesson teaches you how to efficiently "pack" and "ship" massive amounts of information, just like running a large fresh grocery supermarket (FreshCart).

1. Why Learn This?

Imagine you are running an online fresh grocery supermarket (FreshCart), similar to Amazon Fresh. With millions of orders every day, you need to interact with multiple systems such as logistics, payment, and inventory.

If information is not "packed" well or the "shipping" method is wrong, various problems will arise: shipping costs (network bandwidth) are too expensive, systems don't understand each other, or a slight change in the system causes a total crash. This lesson teaches you how to solve these problems to keep your "supermarket" running efficiently.

  • Save Money and Fast: Packing data into tiny binary formats can significantly save network bandwidth and server costs.
  • Cross-language Communication: Different systems may use different programming languages (Python, Java, Go). We need a "universal translator" so they can understand each other.
  • Future-Proofing: Business always changes, and systems need upgrades. We want to ensure that when adding new features, the old systems do not crash—this is called "schema evolution".

The Golden Rule of Data Engineering

"Data needs encoding to move between processes without shared memory."

Because two computers cannot "read minds," we must translate "thoughts" into "written language" (encoded bytes) and then "mail" them out (data flow).

2. Different Tools, Different Ways to "Pack" and "Ship"

A. "Packing Materials": Data Encoding

Textual Formats (JSON, XML, CSV)

Like transparent plastic boxes. Humans can see what's inside at a glance, but they take up a lot of space and are slower for machines to process. Suitable for human reading and simple data exchange.

Learn More
  • JSON (JavaScript Object Notation): A common format for web and mobile apps, lightweight and easy to read.
  • XML (Extensible Markup Language): More complex structure, often used in legacy systems or specific industry standards.
  • CSV (Comma Separated Values): Simple tabular data, separated by commas.

Row-based Binary (Thrift, Protobuf, Avro)

Like vacuum compression bags. Compress data to the extreme, small in size, and fast to transmit. But humans cannot read it; machines need a "vacuum pump" (Schema) to unpack it. Suitable for efficient data transmission within systems.

Learn More
  • Thrift (Apache Thrift): Developed by Facebook, includes an RPC framework, efficient transmission via "field tags".
  • Protobuf (Protocol Buffers): Developed by Google, a purer serialization tool, also uses "field tags".
  • Avro (Apache Avro): Places the "manual" (Schema) directly in the data file header, particularly suitable for big data storage and evolution.

These formats all use pre-defined "contracts" (Schemas) to ensure data structure consistency between different systems and support "schema evolution," meaning upgrading data structures without service interruption.

Columnar Binary (Parquet, ORC, Arrow)

Like categorized supermarket shelves. If you only want to buy a newspaper, you don't need to walk through the entire supermarket. Columnar storage lets you jump straight to the "newspaper section," which is extremely fast. Suitable for big data analysis, reading only the columns needed for the query.

Learn More
  • Parquet (Apache Parquet): The most popular columnar storage format in the big data field, with fast query speeds and high compression rates.
  • ORC (Optimized Row Columnar): Another efficient columnar storage format, similar to Parquet.
  • Arrow (Apache Arrow): In-memory columnar data format, used to accelerate data processing and exchange.

When your boss only wants to see "total sales for last month," columnar storage only needs to read the "amount" column, saving significant time and computing resources.

B. "Shipping Services": Data Flow

REST API & GraphQL (Making a Call / Finding a Personal Shopper)

REST API: Like making a call to customer service. You ask a question, and the agent gives you an answer. This is the most common way for web pages and apps to communicate.
GraphQL: Like finding a personal shopper. You can tell the shopper exactly what you want (e.g., "just give me the user's name and avatar"), no more, no less, saving communication costs.

RPC (gRPC) (Direct Walkie-Talkie Connection)

Like a direct walkie-talkie connection. Different departments within a company (e.g., warehousing and delivery) use "walkie-talkies" to shout for maximum speed. This method is much faster than calling, with almost no latency, and is commonly used for efficient communication between microservices.

Message Queues & Publish/Subscribe (Kafka, RabbitMQ) (Leaving a Note / Loudspeaker Broadcast)

This is an asynchronous shipping method.
Message Queues: Like leaving a note. The front desk receives an order, writes a note, sticks it on the wall, and goes back to other tasks. Warehouse staff see the note and then fulfill the order. The front desk doesn't wait for a response from the warehouse.
Publish/Subscribe: Like a loudspeaker broadcast. A user clicks the "buy" button, and the system shouts over the loudspeaker, "New order received!" All interested departments (inventory, logistics, recommendation systems) hear it and process it independently. This method is very flexible; even if one department is busy, it won't affect the work of others.

3. Real-World Application Scenarios

Business Dashboards (Apache Parquet + SQL)

Scenario: A CEO wants to view the total sales for 2023 from 5 billion transaction records.

Why use Parquet? Because Parquet is columnar storage, the system only needs to read the "sales" column, completely ignoring other columns like "customer name" or "address." This shortens the query from 10 minutes to 2 seconds.

Mobile Apps in Poor Network Environments (GraphQL + JSON)

Scenario: You open a social media app on a 3G network. The app only needs to display the user's avatar and name.

Why use GraphQL? GraphQL allows the phone to tell the server precisely, "just give me the name and photo." This avoids downloading the user's entire history, bio, and settings (which REST might do), saving data and loading faster.

Uber & Netflix Backend (gRPC + Protobuf)

Scenario: You call an Uber. The app must communicate instantly with the driver-matching system, pricing system, and mapping system simultaneously.

Why use Protobuf/gRPC? These internal microservices need to communicate millions of times per second. Protobuf compresses data into tiny binary sizes, and gRPC sends it at extreme speeds without the "bloat" of normal web traffic.

E-commerce Real-time Tracking (Kafka + Avro)

Scenario: On a site like Amazon, every time a user clicks an item, adds to cart, or buys, multiple systems need to know (fraud detection, inventory, recommendation engine).

Why use Kafka Pub/Sub + Avro? The website "publishes" click events to Kafka. Fraud detection and inventory systems "subscribe" to these click events and monitor them. Avro is used because it perfectly handles changing data structures (for example, if Amazon later adds a new "color" attribute to click events).
